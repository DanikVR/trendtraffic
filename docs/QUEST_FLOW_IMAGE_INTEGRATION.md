# Quest Flow → VibeVox: преобразование изображений (Фаза 1)

Документ для разработчиков **Quest Flow**. Описывает, как кнопки Telegram-бота должны
вызывать VibeVox, чтобы сгенерировать/обработать изображение и вернуть его клиенту.

> Фаза 1 — **только изображения** (модель Nano Banana / Gemini Image). Видео не реализуется.
> Всё, что бот делает сейчас (текст, распознавание фото/видео/голоса), работает **без изменений**.

---

## 1. Идея

1. Владелец в VibeVox создаёт **блоки обработки** (Настройки → Quest Flow → «Блоки обработки изображений»).
   У каждого блока есть **ключ пресета** (`preset_key`), например `remove_object`, `tryon_jacket`.
2. В Quest Flow строится меню/кнопки бота. Кнопка ведёт в ветку, которая вызывает VibeVox
   и **передаёт `preset_key`** этого блока + изображение(я) клиента.
3. VibeVox берёт **ровно этот** пресет (ИИ ничего не выбирает), генерирует/правит картинку и
   возвращает её URL в ответе. Quest Flow отправляет картинку клиенту (sendPhoto).

**Жёсткая привязка = `preset_key` в теле запроса.** Без него запрос обрабатывается как обычный
AI-диалог (как сейчас).

---

## 2. Авторизация

Как и у существующего вебхука — заголовок:

```
Authorization: Bearer <ваш ключ vbvx_qf_...>
```

(Ключ берётся в Настройки → Quest Flow → «API-ключи Quest Flow».)

---

## 3. Эндпоинты

База: `https://<домен VibeVox>/api/quest-flow`

> Требование: на сервере VibeVox задан `PUBLIC_BASE_URL` (или `APP_BASE_URL`) — иначе ссылка
> на картинку придёт относительной и облако Telegram её не скачает.

### 3.1. JSON — фото в base64 или генерация по тексту

`POST /api/quest-flow/inbound`  ·  `Content-Type: application/json`

| Поле | Тип | Обяз. | Описание |
|------|-----|-------|----------|
| `telegram_bot_id` | string | да | ID бота |
| `telegram_user_id` | string | да | chat_id клиента |
| `preset_key` | string | **да (для картинок)** | ключ блока обработки |
| `text` | string | — | подпись/запрос клиента (что нарисовать/изменить) |
| `image` | string(base64) | — | фото клиента (без префикса `data:`) |
| `image_mime` | string | — | напр. `image/jpeg` |
| `images` | array | — | несколько картинок клиента (перенос стиля A+B, коллаж): `[{ "data": "<base64>", "mime": "image/jpeg" }, …]` |
| `telegram_username` | string | — | для отображения у владельца |

### 3.2. Multipart — крупный файл (рекомендуется для фото из Telegram)

`POST /api/quest-flow/inbound-media`  ·  `Content-Type: multipart/form-data`

| Поле формы | Обяз. | Описание |
|------------|-------|----------|
| `file` | да | сам файл изображения |
| `telegram_bot_id` | да | ID бота |
| `telegram_user_id` | да | chat_id клиента |
| `preset_key` | **да (для картинок)** | ключ блока обработки |
| `text` (или `caption`) | — | подпись/запрос клиента |
| `telegram_username` | — | для отображения |

## 4. Ответ

```jsonc
{
  "ok": true,
  "roomId": "…",
  "response": {
    "text": "Готово!",                                   // подпись (может быть пустой)
    "imageUrl": "https://ваш-домен/uploads/enterprise-chat/qf-gen-xxxx.png", // АБСОЛЮТНЫЙ URL
    "imageMime": "image/png"
  },
  "balanceRemaining": -1
}
```

**Что делает Quest Flow:** если `response.imageUrl` не пустой → отправить клиенту **фото** по этому
URL (Telegram `sendPhoto`), подпись = `response.text`. Если `imageUrl` пустой, а `response.text`
не пустой → отправить текст (это сообщение об ошибке/уточнение — генерация не удалась).

> Генерация может занимать до ~2 минут (особенно Nano Banana Pro / несколько картинок) —
> таймаут HTTP-блока в Quest Flow ставьте **≥ 120 c**.

## 5. Какие входы нужны функциям

Зависит от функции блока (видно владельцу в UI). Сводка:

| Функция | Фото клиента | Доп. вход | Откуда «исходная» картинка |
|---------|--------------|-----------|----------------------------|
| Генерация с нуля | нет | — | только `text` |
| Правка изображения | 1 фото | — | фото клиента |
| Локальная правка | 1 фото | — | фото клиента |
| Удаление/добавление объекта | 1 фото | — | фото клиента |
| Композиция / вставка | 1 фото | референс(ы) у пресета | фото клиента + референсы |
| Коллаж / объединение | несколько (`images`) | можно референсы | все приложенные изображения |
| Виртуальная примерка | 1 фото | референс у пресета | фото клиента + референс |
| Перенос стиля | 1 фото (A) | стиль B (референс или 2-е фото в `images`) | A + B |
| Брендирование | 1 фото | логотип (референс) | фото клиента + референс |
| Консистентность персонажа | нет | референс-лицо у пресета | `text` + референс |
| Итеративная правка | нет | — | последняя сгенерированная картинка комнаты |
| Текст внутри картинки | нет | — | только `text` |

«Референс у пресета» = фикс. картинка (лого/товар/стиль), которую загрузил владелец в блоке;
её слать из Quest Flow **не нужно** — она уже привязана к пресету.

### Сколько картинок можно слать (лимит по модели)

Суммарно (фото клиента в `images` + референсы пресета) на один запрос:

| Модель (выбирается в Настройках) | Рекомендуемый максимум |
|----------------------------------|------------------------|
| Nano Banana Pro (`gemini-3-pro-image`) | **до 14** |
| Nano Banana / Nano Banana 2 (flash) | **~3–5** (дальше качество падает) |

Жёсткий потолок на стороне VibeVox — **14**; лишние сверх него отбрасываются. Для коллажей из
многих фото ставьте модель **Nano Banana Pro**.

## 6. Примеры

### 6.1. Удаление объекта (кнопка «Убрать лишнее»)

Бот просит фото + текст «что убрать», затем:

```bash
curl -X POST https://ВАШ-ДОМЕН/api/quest-flow/inbound-media \
  -H "Authorization: Bearer vbvx_qf_XXXX" \
  -F "telegram_bot_id=123456" \
  -F "telegram_user_id=99887766" \
  -F "preset_key=remove_object" \
  -F "text=убери провода на фоне" \
  -F "file=@photo.jpg"
```

### 6.2. Генерация с нуля (кнопка «Нарисовать»)

```bash
curl -X POST https://ВАШ-ДОМЕН/api/quest-flow/inbound \
  -H "Authorization: Bearer vbvx_qf_XXXX" \
  -H "Content-Type: application/json" \
  -d '{
    "telegram_bot_id": "123456",
    "telegram_user_id": "99887766",
    "preset_key": "logo_from_scratch",
    "text": "минималистичный логотип кофейни, бежево-коричневый"
  }'
```

### 6.3. Перенос стиля — два изображения A и B (JSON, base64)

```bash
curl -X POST https://ВАШ-ДОМЕН/api/quest-flow/inbound \
  -H "Authorization: Bearer vbvx_qf_XXXX" \
  -H "Content-Type: application/json" \
  -d '{
    "telegram_bot_id": "123456",
    "telegram_user_id": "99887766",
    "preset_key": "style_transfer",
    "text": "перерисуй фото A в стиле B",
    "images": [
      { "data": "<base64 фото A>", "mime": "image/jpeg" },
      { "data": "<base64 стиль B>", "mime": "image/jpeg" }
    ]
  }'
```

### 6.4. Виртуальная примерка (товар — фикс. референс пресета)

Клиент шлёт только своё фото; товар уже привязан к пресету в VibeVox:

```bash
curl -X POST https://ВАШ-ДОМЕН/api/quest-flow/inbound-media \
  -H "Authorization: Bearer vbvx_qf_XXXX" \
  -F "telegram_bot_id=123456" \
  -F "telegram_user_id=99887766" \
  -F "preset_key=tryon_jacket" \
  -F "file=@client_photo.jpg"
```

### 6.5. Коллаж из нескольких изображений (JSON, base64)

Клиент присылает несколько фото — собираем в одну картинку (лучше на модели Nano Banana Pro):

```bash
curl -X POST https://ВАШ-ДОМЕН/api/quest-flow/inbound \
  -H "Authorization: Bearer vbvx_qf_XXXX" \
  -H "Content-Type: application/json" \
  -d '{
    "telegram_bot_id": "123456",
    "telegram_user_id": "99887766",
    "preset_key": "collage",
    "text": "собери коллаж 2x2 из этих фото",
    "images": [
      { "data": "<base64 фото 1>", "mime": "image/jpeg" },
      { "data": "<base64 фото 2>", "mime": "image/jpeg" },
      { "data": "<base64 фото 3>", "mime": "image/jpeg" },
      { "data": "<base64 фото 4>", "mime": "image/jpeg" }
    ]
  }'
```

## 7. Сессии/разделы (например «Виртуальная примерочная»)

Если клиент «входит в раздел» и общается внутри него — Quest Flow держит состояние ветки и на
**каждый** вызов шлёт один и тот же `preset_key` этого раздела. VibeVox остаётся без состояния.
Для функции «Итеративная правка» VibeVox сам берёт последнюю сгенерированную картинку комнаты.

---

## 8. Ошибки

| Код | Значение |
|-----|----------|
| 401 | нет/невалиден `Authorization: Bearer` |
| 402 | у аккаунта не тариф Enterprise |
| 400 | не переданы обязательные поля |
| 500 | внутренняя ошибка |

Если сама генерация не удалась (фильтр модели/ошибка) — вернётся `200` с `imageUrl: null` и
дружелюбным `response.text` (его и отправьте клиенту).

## 9. Реализация на стороне Quest Flow (v1.6.0)

> Зафиксировано по отчёту команды Quest Flow. Контракт подтверждён и совместим с разделами 1–8.
> На стороне VibeVox этот контракт уже принимается (поле `image` base64 в JSON `/inbound`,
> `preset_key`, `metadata` — обрабатываются).

### 9.1. Блок VibeVox теперь работает в двух режимах одновременно
- **Фоновый диалог** (как раньше) — отвечает на любые свободные сообщения игрока
  (текст/голос/фото/видео). Без изменений.
- **Кнопки-функции** (новое) — внутри блока добавляются цветные кнопки (как в блоке
  «Сообщение»). Каждая жёстко привязана к `preset_key` функции VibeVox и имеет
  **индивидуальный выход** — после обработки картинки поток идёт по нему дальше. Блок можно
  ставить в цепочку сколько угодно раз. Блок без кнопок = чистый фоновый passthrough.

**Поля кнопки в редакторе:** Текст · Цвет (default/синий/зелёный/красный) · Анимир. смайлик
(`icon_custom_emoji_id`) · `preset_key` (владелец вписывает вручную) · Режим (Фото/Текст) ·
Сообщение игроку при нажатии.

### 9.2. Поток выполнения
```
[Игрок] жмёт кнопку → callback_data = vvbtn:{nodeId}:{btnId}
QF: ставит ожидание ввода, пишет «Пришлите фото / Опишите…»
[Игрок] присылает фото (режим «Фото») ИЛИ текст (режим «Текст»)
QF: «⏳ Подождите 2–3 минуты…» → POST /api/quest-flow/inbound (preset_key + image/text)
[VibeVox] → response.imageUrl
QF: sendPhoto (подпись = response.text) → поток идёт по ИНДИВИДУАЛЬНОМУ выходу кнопки
Фоновый VibeVox-диалог продолжает работать.
```
- Только `response.text` без `imageUrl` → QF отправляет текст (ошибка/уточнение).
- Сетевая ошибка/таймаут → дружелюбное сообщение, цепочку НЕ продвигаем (кнопки остаются —
  можно повторить).

### 9.3. Что Quest Flow отправляет (для обоих режимов — JSON `/inbound`)
- **Фото:** `{ telegram_bot_id, telegram_user_id, preset_key, text?, image: "<base64 без data:>",
  image_mime, telegram_username, metadata: { node_id, preset_key, quiz_id } }`
- **Текст:** `{ telegram_bot_id, telegram_user_id, preset_key, text }`
- Таймаут HTTP на стороне QF для preset-функций — **120 c**.

### 9.4. Фаза 2 (несколько изображений)
Коллаж / перенос стиля A+B (`images[]`) + референсы-у-пресета — следующий этап. Клиент QF уже
поддерживает `images[]` (`VibevoxInboundInput.imagesBase64`), останется добавить UI-сбор
нескольких фото. На стороне VibeVox `images[]` уже принимается (см. §3.1 и §6.5).
