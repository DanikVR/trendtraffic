# Модуль «Двойные задачи + авто-сообщения» (CRM/Канбан) — интеграция с Chatwoot

> ТЗ + руководство, **адаптированное под стек и архитектуру VibeVox**.
> Источник — спецификация владельца (дополненная версия с двумя типами задач).
> Этот документ — план; реализация поэтапная (см. §«Фазы»).

---

## 0. Что это и почему «в нашем стиле»

Модуль управления задачами по клиенту, **встроенный в правую панель Chatwoot** (через
Dashboard Apps / iframe) + полноэкранный **Канбан/Календарь** в кабинете VibeVox.

**Две принципиально разные задачи на контакт (бесконечное число):**
- **`internal_notification` — «Напомнить оператору».** Внутреннее напоминание (клиент не
  видит). В `due_at` → пуш оператору (звук+тост в кабинете + FCM на телефон) + deep-link в диалог.
- **`client_message` — «Написать клиенту».** В `due_at` бэкенд сам шлёт готовый текст клиенту
  через Chatwoot API (в его канал) и помечает задачу выполненной. Оператор может спать.

**Адаптация под VibeVox (vs. исходная спека):**
| Спека | У нас (VibeVox) |
|---|---|
| Next.js / Drizzle ORM | **React + Vite + Tailwind** (фронт VibeVox) · **Express + `pg`** (бэк VibeVox), миграции как в `db/migrations.ts` |
| `saas_account_id` (отдельная SaaS) | **tenant VibeVox** (`tenants.id`) — уже есть жёсткая изоляция; маппинг на Chatwoot-аккаунт уже храним (`tenants.chatwoot_account_id`, см. CW-SSO мост) |
| Контекст через query-параметры URL | **postMessage Dashboard App** (безопаснее) + подписанный токен (см. §Безопасность) |
| BullMQ+Redis (жёстко) | MVP — **планировщик на БД+таймерах** (без новой инфры); масштаб — BullMQ+Redis опц. |
| Отправка клиенту «в любой канал» | то же **Chatwoot Application API**, НО ⚠️ учёт **окна 24ч** Meta (см. §Анализ) |

# 🛠 ЧАСТЬ 1. ТЕХНИЧЕСКОЕ ЗАДАНИЕ

## 1. Архитектура и контекст

```
┌──────────────── Chatwoot (форк, app.vibevox.pro) ─────────────────┐
│  Диалог → правый сайдбар: Dashboard App (iframe)                  │
│     ▼ postMessage: {conversation, contact, currentAgent, account} │
│  Кнопка «+ Поставить задачу» в шапке контакта (патч форка)        │
└───────────────────────────┬───────────────────────────────────────┘
                            │ iframe src + signed token
                            ▼
┌──────────── VibeVox embed SPA  (my.vibevox.pro/embed/kanban-sidebar) ─┐
│  React+Tailwind, стиль VibeVox; список задач + модалка создания      │
└───────────────────────────┬───────────────────────────────────────────┘
                            │ /api/tasks (Bearer signed token)
                            ▼
┌──────────── VibeVox backend (Express + pg) ───────────────────────────┐
│  CRUD задач · планировщик (due_at) · воркер:                          │
│   • internal_notification → WS-тост + FCM-пуш оператору               │
│   • client_message → POST в Chatwoot API → доставка клиенту           │
└────────────────────────────────────────────────────────────────────────┘
```

**Контекст НЕ берём из голых query-параметров** (их можно подделать). Берём двумя путями:
1. **postMessage Dashboard App** — Chatwoot сам отдаёт iframe объект `{conversation, contact,
   currentAgent, currentAccount}` по запросу `window.parent.postMessage('chatwoot-dashboard-app:fetch-info', '*')`.
   Это штатный безопасный механизм (источник проверяем по origin).
2. **Подписанный токен** в URL Dashboard App (`?cwt=<JWT>`), сгенерённый VibeVox при выдаче
   Dashboard App тенанту — несёт `tenant_id` + подпись (HMAC `JWT_SECRET`). Бэкенд проверяет его
   на КАЖДОМ запросе и НЕ доверяет `account_id`/`contact_id` с клиента.

## 2. Схема БД (PostgreSQL, миграция VibeVox)

```sql
-- идемпотентно, в стиле apps/backend/src/db/migrations.ts
CREATE TYPE task_type   AS ENUM ('internal_notification', 'client_message');
CREATE TYPE task_status AS ENUM ('todo', 'completed', 'failed');

CREATE TABLE IF NOT EXISTS crm_tasks (
  id                        UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
  tenant_id                 UUID NOT NULL REFERENCES tenants(id) ON DELETE CASCADE, -- изоляция VibeVox
  chatwoot_account_id       VARCHAR(64)  NOT NULL,   -- для Chatwoot API
  chatwoot_contact_id       VARCHAR(64)  NOT NULL,   -- привязка к контакту
  chatwoot_conversation_id  VARCHAR(64),             -- диалог (для сообщения/deep-link)
  assigned_operator_id      VARCHAR(64)  NOT NULL,   -- оператор-исполнитель (Chatwoot user id)
  type                      task_type    NOT NULL,
  body                      TEXT         NOT NULL,   -- текст напоминания ИЛИ сообщения клиенту
  due_at                    TIMESTAMPTZ  NOT NULL,
  status                    task_status  NOT NULL DEFAULT 'todo',
  fired_at                  TIMESTAMPTZ,             -- когда сработала (аудит)
  error                     TEXT,                    -- если status='failed'
  created_at                TIMESTAMPTZ DEFAULT now(),
  updated_at                TIMESTAMPTZ DEFAULT now()
);
CREATE INDEX IF NOT EXISTS idx_crm_tasks_contact ON crm_tasks(tenant_id, chatwoot_contact_id, status);
CREATE INDEX IF NOT EXISTS idx_crm_tasks_due ON crm_tasks(due_at) WHERE status = 'todo'; -- для планировщика
```
⚠️ В **fallback-режиме** (без Postgres) — добавить обработчики в `db/index.ts` ([[project_fallback_durability]]). На проде — реальный Postgres.

## 3. API (Express, `/api/crm-tasks`, JWT+Enterprise)

| Метод | Путь | Что |
|---|---|---|
| `GET` | `/api/crm-tasks?contact_id=` | задачи по контакту: сначала `todo` по возр. `due_at`, потом архив `completed`. |
| `POST` | `/api/crm-tasks` | создать. Body: `type, body, due_at, assigned_operator_id, chatwoot_contact_id, chatwoot_conversation_id`. `tenant_id`/`chatwoot_account_id` берём из **подписанного токена**, НЕ с клиента. |
| `PATCH` | `/api/crm-tasks/:id` | сменить `status` (todo↔completed) / перенести `due_at`. Пере-регистрирует таймер. |
| `DELETE`| `/api/crm-tasks/:id` | удалить + снять таймер. |
| `GET` | `/api/crm-tasks/board?days=&operator_id=` | все задачи аккаунта для Канбан/Календаря (фильтры). |

Авторизация: каждый запрос проверяет подписанный токен (tenant_id) + что `contact_id`/`conversation_id`
принадлежат этому tenant/account (свериться по нашему маппингу / Chatwoot API). **Никогда не доверять
`account_id` с клиента.**

## 4. Планировщик «минута в минуту»

**MVP (без новой инфры) — БД + точные таймеры:**
1. При `POST`/`PATCH` бэкенд считает `delay = due_at - now()`.
2. Если задача наступает в ближайшие ~10 мин — ставим точный `setTimeout` в процессе.
3. Cron-тик раз в минуту подбирает задачи `todo` с `due_at <= now()+10мин` (на случай рестарта/дальних) и ставит таймеры.
4. На рестарте бэкенда — reload: все `todo` будущие перезаряжаются (источник истины — БД).
5. Срабатывание (воркер): перечитать статус из БД (если уже `completed`/`deleted` — отмена), иначе выполнить по `type` (§5).

**Масштаб (опц.) — BullMQ + Redis:** delayed jobs с `jobId = task.id`; при изменении — `remove`+`add`.
Redis на проде уже есть (стек Chatwoot) — можно поднять отдельный инстанс для VibeVox.

## 5. Логика срабатывания по типу

**`internal_notification` (напомнить оператору):**
1. WebSocket-событие в сессию `assigned_operator_id` → в кабинете VibeVox всплывает тост со звуком.
2. FCM-пуш на телефон оператора. Deep-link: `https://app.vibevox.pro/app/accounts/{chatwoot_account_id}/conversations/{chatwoot_conversation_id}` → клик открывает нужный диалог.
3. Статус НЕ меняется автоматически — оператор сам жмёт галочку, когда выполнил.

**`client_message` (написать клиенту):**
1. Бэкенд POST в Chatwoot Application API (тем же админ-токеном, что и CW-SSO мост):
   `POST /api/v1/accounts/{chatwoot_account_id}/conversations/{conv_id}/messages`
   `{ "content": "<body>", "message_type": "outgoing" }`.
2. Chatwoot сам доставляет в канал клиента (Telegram/WhatsApp/…).
3. Успех → `status='completed'`, `fired_at=now()`. Ошибка → `status='failed'`, `error=...` (+ ретрай-политика).

## 🔍 АНАЛИЗ И КРИТИЧНЫЕ УТОЧНЕНИЯ (моё «доработай в нашем стиле»)

1. **⚠️ Окно 24 часа (Meta) — главный подводный камень `client_message`.** На WhatsApp/Instagram/
   Facebook **нельзя** слать свободный текст позже 24ч от последнего сообщения клиента — только
   одобренные **шаблоны** (HSM). Сценарий «напишу клиенту завтра в 10:00» в WhatsApp **упадёт**,
   если прошло >24ч. → **Telegram/SMS/Email — без окна** (работает всегда); **WA/IG/FB** — в UI
   показывать предупреждение и при `due_at` вне окна слать **шаблон** (или запретить свободный
   текст). Это согласуется с [[project_omnichannel_plan]] (окно 24ч).
2. **🔒 Безопасность iframe.** Голые query-параметры из спеки = дыра в мультиарендности (любой
   подменит `account_id`). → только **postMessage Dashboard App + подписанный токен** (§1), все
   запросы фильтруем по проверенному `tenant_id`.
3. **🔗 Маппинг tenant ↔ Chatwoot account уже есть** — `tenants.chatwoot_account_id` + таблица
   `chatwoot_users` (CW-SSO мост). Переиспользуем: админ-токен для отправки берём оттуда, не плодим.
4. **🧩 Кнопка «+ задача» в шапке контакта = патч форка Chatwoot** (зона презентации). Документировать
   в `VIBEVOX_BRANDING` + не трогать коннекторы (см. `UPSTREAM_SYNC.md`). Сам iframe-сайдбар — штатные
   Dashboard Apps, патч НЕ нужен.
5. **🔁 Идемпотентность срабатывания.** Воркер перед действием перечитывает статус (анти-дубль при
   рестартах/гонках). Для `client_message` — анти-двойная-отправка (флаг `fired_at` + транзакция).
6. **🌍 i18n.** Тосты/кнопки/модалка — через i18next VibeVox (108 языков), не хардкодить.
7. **📞 FCM — отдельная инфра** (Firebase проект + service worker + хранение device-токенов оператора).
   Для MVP можно начать только с WS-тоста в кабинете, FCM добавить фазой 2.

# 👥 ЧАСТЬ 2. РУКОВОДСТВО (операторам и руководителям)

Система объединяет общение с клиентами в мессенджерах и личные задачи — без сторонних CRM и блокнотов.
На одного клиента — сколько угодно задач, все видны списком в правой панели чата. Оператор выбирает один из двух сценариев.

**Вариант 1 — «Внутреннее напоминание оператору».** Когда нужно спланировать действие для себя
(клиент не видит): проверить оплату, согласовать скидку, подготовить документы.
- Жмёшь «Поставить задачу» → вкладка «Напомнить оператору» → дата/время → текст для себя → Сохранить.
- В назначенное время клиенту ничего не уходит, а у оператора (на ПК и телефоне) — звук + пуш
  «🚨 Дедлайн: …». Клик по пушу → сразу открывается нужный диалог.

**Вариант 2 — «Авто-сообщение клиенту».** Когда нужно автоматически отправить готовый текст в
мессенджер клиента ровно в срок, без участия человека.
- Жмёшь «Поставить задачу» → вкладка «Написать клиенту» → дата/время → текст для клиента → Сохранить.
- В срок система сама шлёт сообщение в канал клиента. Задача отмечается выполненной.
- ⚠️ В WhatsApp/Instagram, если с момента последнего сообщения клиента прошло больше суток —
  отправка возможна только одобренным шаблоном (ограничение мессенджеров, не наше). В Telegram — без ограничений.

**Контроль выполнения.**
- *Оператор (в чате):* справа список задач, у каждой чекбокс. Внутренние — отмечаешь галочкой сам;
  авто-сообщения система зачёркивает сама после отправки. Просроченные (`todo` + время прошло) —
  красные, вверху списка.
- *Руководитель (кабинет SaaS):* Канбан-доска / Календарь — все задачи по часам, фильтр по
  оператору; видно сколько закрыто, сколько авто-сообщений запланировано, что просрочено (красный
  блок). С карточки — «Перейти в чат».
- *Изоляция:* каждый клиент платформы видит только свой кабинет/операторов/задачи — строгая
  мультиарендность, пересечений между компаниями нет.

## ✅ Статус реализации (2026-06-01)

**Фаза 0 + бэкенд Фазы 1 — СДЕЛАНО** (монолит VibeVox, `tsc` чистый, CRUD проверен вживую):
- Миграция `crm_tasks` (изоляция `tenant_id` + `ON DELETE CASCADE`), индексы.
- API `/api/crm-tasks` (CRUD, JWT+Enterprise, фильтр по tenant).
- БД-таймер планировщик (раз в минуту): `internal_notification` → событие `crmTaskEvents` (WS на фронте);
  `client_message` → авто-отправка через Chatwoot API → `completed`/`failed`.

**Конструктивные замечания владельца — учтены в коде:**
1. ✅ **Валидация прошлого:** `due_at` в прошлом → `400` (бэкенд-страховка; фронт тоже блокирует DatePicker).
2. ✅ **UTC:** бэкенд нормализует в `new Date().toISOString()` (строго UTC под `TIMESTAMPTZ`); **контракт фронта — слать ISO 8601 UTC** (DatePicker оператора в Варшаве → UTC на сервер).
3. ✅ **Каскад при удалении контакта в Chatwoot:** вебхук `contact_deleted` → `deleteTasksForChatwootContact(account_id, contact_id)`. Доп. страховка: `client_message` несуществующему контакту → Chatwoot вернёт ошибку → задача `failed`, а не «в пустоту».

Файлы: `db/migrations.ts`, `modules/crm_tasks/{service,router}.ts`, `server.ts`, `modules/channels/chatwoot_router.ts`.

## ✅ Решения владельца (ФИНАЛ, 2026-06-01)
1. **Где живёт модуль:** внутри **монолита VibeVox** (Express + pg + React/Vite). Отдельных сервисов нет.
2. **Очередь:** **БД-таймер** — cron/интервал в Express раз в минуту: `SELECT … WHERE due_at <= NOW() AND status='todo'`. Никакого BullMQ/Redis на старте.
3. **FCM:** на **Фазу 3**. MVP — только WS-тост со звуком в интерфейсе оператора.

5. **Цепочки авто-сообщений:** несколько задач наперёд на один чат (пример «клиника»: напоминание за день → в день → опрос после приёма). Работает из коробки (бесконечно задач на контакт).

**Интерфейс — строго в стиле VibeVox, «единая программа».**
